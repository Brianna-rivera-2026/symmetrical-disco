1. Process Topology & Docker Orchestration

Five compose services, one shared image (jobprocessor-app:${APP_TAG:-dev}) built once and reused via the image: key on all app services — API, worker, ticker, and migrate differ only by command.

Q: Why is the graceful shutdown chain correct?
SIGTERM → signal handler flips a shutting_down flag → the worker loop checks _should_stop() only between messages, so the in-flight job finishes, gets committed and XACKed, then the loop exits. Compose waits stop_grace_period: 50s before SIGKILL; 50s > 45s handler timeout, so even the slowest legal handler completes. The signal handler is registered in run_forever (app/worker/runner.py), not in __main__.
The flag is a dict (shutting_down = {"flag": False}) not a bare bool — a closure can't rebind an outer local without nonlocal; mutating a dict sidesteps that. An interviewer may ask why it's not a threading.Event: signal handlers run on the main thread and the loop is single-threaded, so a plain mutable cell is sufficient.

2. Job Pickup — Preventing Duplicate Execution

app/queue/consumer.py · app/repository.py:claim_job · app/worker/runner.py:process_job

Two independent layers guarantee a job executes at most once at a time:

  1. Redis layer (delivery): XREADGROUP in a consumer group assigns each stream entry to exactly one consumer and tracks it in the Pending Entries List (PEL) until XACK.
  2. Postgres layer (execution — the real guard): an atomic compare-and-swap: 
    UPDATE jobs SET status='processing', started_at=now()
WHERE id=:id AND status IN ('pending','scheduled')
    claim_job returns result.rowcount == 1. Only one actor can ever win because Postgres row-level locking serializes the two UPDATEs; the loser sees rowcount 0, logs job.skipped reason=not_claimable, and ACKs the message anyway (it's a duplicate delivery, not a job to run).
Q: Why both layers? Isn't the consumer group enough?
No. Duplicate deliveries are legal in this design: the ticker's crash-safe promotion (XADD before ZREM, §6) can double-enqueue; the reaper re-enqueues after XAUTOCLAIM; the reconciler re-enqueues unsynced rows. The DB claim-guard converts "at-least-once delivery" into "at-most-once concurrent execution". The stream is treated as a dispatch signal; Postgres is the source of truth.
Q: Why does claim_job accept scheduled as well as pending?
The ticker promotes in two steps (Redis XADD, then the DB status flip to pending). If a worker is fast enough to read the message before the DB update lands, the row is still scheduled. Accepting both states closes that window without ordering constraints. Covered by test_claim_accepts_scheduled_state.

Consumer identity

CONSUMER_NAME = f"worker_{os.getenv('HOSTNAME','local')}_{uuid.uuid4().hex[:6]}"

Unique per process start, not per container: a restarted worker gets a fresh name, so its old PEL entries go stale and are reclaimed by the reaper rather than being silently inherited. HOSTNAME gives operator-readable attribution in XINFO CONSUMERS.

Group creation

ensure_group uses XGROUP CREATE ... id="$" MKSTREAM and swallows only BUSYGROUP (idempotent). Because id="$" means "only messages after group creation", every producer creates the groups before its first XADD — API lifespan, worker startup, and ticker startup all call ensure_group; the ticker explicitly comments that this prevents promoted jobs from being invisible to a group created later.

3. Ack Semantics — the Outcome / "won" Protocol

app/worker/runner.py · app/retry.py

process_job returns Outcome(ack, recycle, label). The rule everywhere: XACK only if you won the guarded DB transition (every state-changing repository function returns rowcount == 1). Ack always happens after the DB commit — ack-after-commit ordering means a crash between commit and ack yields a redelivery that the claim/status guards absorb, never a lost job.

PathackWhy
Claim lost (skipped)trueDuplicate delivery; someone else owns the job.
Completed, wontrueNormal case.
Completed, lost (lost)falseThe reaper stole the job mid-flight (worker exceeded visibility timeout but still finished). The worker logs job.complete_lost_to_reaper at critical and leaves the message in the PEL — the reaper will later see the row's terminal/processing status and resolve it. Acking here would hide the double-execution signal.
Retried / timed out / cancelledwonIf this worker lost the guarded transition, another actor (reaper) owns the message's fate; the loser must not ack.
This "lost to reaper" branch is the honest answer to "can a job run twice?" — yes, if a handler outruns visibility_timeout_s while its process is alive-but-slow and the reaper reclaims it. The system is at-least-once; mitigations are the config invariant job_handler_timeout_s (45) < visibility_timeout_s (60) (validated by a Pydantic model_validator that refuses to boot otherwise) plus the critical log. It is not exactly-once and the docs never claim it is.

4. The is_synced_to_redis Flag — a Lightweight Outbox

app/repository.py:mark_synced, list_unsynced · app/ticker/runner.py:reconcile_orphans · migration 0002

Postgres and Redis cannot be updated in one transaction, so every enqueue site follows a write-DB-first, then-Redis, then-flag sequence:

1. INSERT/UPDATE job row  (is_synced_to_redis = false)   ← commit
2. XADD to stream  /  ZADD to jobs:delayed
3. UPDATE jobs SET is_synced_to_redis = true              ← commit

A crash after step 1 or 2 leaves the flag false. The ticker's reconcile_orphans sweeps (every reconcile_interval_s=60s) for rows with is_synced_to_redis = false AND status IN ('pending','scheduled') AND created_at < now() - grace and finishes the handoff — pending rows to their priority stream, scheduled rows into the ZSET.

  • Grace period (reconcile_grace_s=10s): avoids racing a live API request that is between steps 1 and 3 — without it, the reconciler could double-enqueue every fresh submission. Double-enqueue is safe anyway (claim guard) but noisy.
  • Partial index: migration 0002 creates ix_jobs_unsynced ON (created_at, id) WHERE is_synced_to_redis = false AND status IN ('pending','scheduled') — the sweep query is index-only against a tiny index no matter how large jobs grows.
  • Backfill nuance: the same migration runs UPDATE jobs SET is_synced_to_redis = TRUE so pre-existing rows are never re-enqueued when the feature ships.
  • Batching: the sweep loops in reconcile_batch_size=500 chunks until a short page, so a huge backlog can't blow memory.
  • Retry transitions (retry_to_pending/retry_to_scheduled) explicitly reset the flag to false — the same reconciler covers crashes in the retry path.
Q: Why not a real transactional outbox table?
A separate outbox table + relay gives the same guarantee with more moving parts; here the job row itself is the outbox record because there is exactly one message per row and the message content is just the job id. The flag column collapses the outbox into the aggregate.
Q: What if scheduled_at is NULL on an unsynced scheduled row?
"Impossible" state, but reconcile_orphans still guards it: logs ticker.reconcile_skipped_null_scheduled_at and skips instead of crashing the tick. Defensive coding against data corruption is a recurring pattern here.

5. Priority — Three Streams, Strict High→Normal→Low

app/queue/consumer.py:read_priority · app/core/config.py · DECISIONS.md §3

Redis Streams have no priority primitive, so priority = which stream: jobs:stream:high|normal|low, chosen once via settings.stream_for_priority(priority) at every enqueue site (submit, retry, promote, reconcile, reap).

The two-phase read algorithm (the subtle part)

# Phase 1: probe each stream highest-first, NON-blocking, count=1
for stream in [high, normal, low]:
    msgs = XREADGROUP ... {stream: ">"} count=1 block=None
    if msgs: return msgs          # higher backlog fully drained first

# Phase 2: all empty → ONE blocking read across all three (block_ms=5000)
return XREADGROUP ... {high: ">", normal: ">", low: ">"} count=1 block=5000
Q: Why not just one blocking XREADGROUP over all three streams all the time?
Because a multi-stream XREADGROUP returns whichever stream has data without draining higher ones first across iterations; the per-stream non-blocking probe guarantees strict drain order. The blocking multi-read is only reached when everything is empty, so it can never reorder a real backlog — and Redis preserves the requested stream order in its reply, keeping priority even for messages arriving during the block.
Q: Cost of this design?
When only low-priority work exists, every message costs 3 round trips (2 empty probes + 1 hit). Accepted: correctness of strict ordering over throughput. Also count=1 everywhere — one job per read maximizes fairness across workers and bounds the blast radius of a worker crash to a single in-flight job.
Q: Starvation?
Deliberate. A sustained high-priority flood starves normal/low indefinitely. DECISIONS.md documents the escape hatch: since workers read settings.ordered_streams, you could dedicate replicas to a restricted stream list (config knob, not implemented).

6. Scheduled Jobs — Delayed ZSET + Ticker Promotion

app/queue/delayed.py · app/ticker/runner.py:promote_due · app/api/routes.py:_create_and_handoff

Future jobs live in a Redis sorted set jobs:delayed with score = epoch seconds of scheduled_at. Submission with a future scheduled_at creates the row as scheduled and ZADDs; a past/absent scheduled_at goes straight to a stream as pending (tested: test_submit_past_scheduled_at_runs_immediately).

Ticker promote loop, per 1s tick

  1. ZRANGEBYSCORE jobs:delayed 0 now LIMIT 0 100 → due ids.
  2. Bulk-fetch priorities from Postgres (get_priorities, single SELECT id, priority WHERE id IN (...)).
  3. Ids with no DB row / no priority → dropped, not enqueued (cancelled or deleted while parked) but still ZREM'd so they can't accumulate forever.
  4. promote(): pipelined XADDs to each id's priority stream before a single multi-member ZREM.
  5. promote_scheduled_to_pending: one bulk UPDATE ... WHERE id IN (...) AND status='scheduled'.
Q: Why XADD before ZREM?
Crash-safety direction: crash mid-promotion leaves ids in the ZSET → retried next tick → possible duplicate stream entries, which the claim guard absorbs. The reverse order (ZREM first) could lose jobs on crash. Chose duplicates over loss — the system-wide theme.
Q: What if 10,000 jobs come due in the same instant?
The drain loop: if a tick promoted a full batch (≥ ticker_batch_size=100) it skips the 1s sleep and loops immediately — 10k jobs drain in ~100 back-to-back batches instead of trickling 100/second for 100s.
Q: Timezone handling?
JobSubmission.scheduled_at has a field validator: naive datetimes are assumed UTC (replace(tzinfo=utc)), aware ones are converted (astimezone(utc)). All DB timestamps are TIMESTAMP(timezone=True); all code uses datetime.now(timezone.utc) — no naive datetimes cross a boundary.

Ticker error handling: the whole tick body is wrapped in try/except Exception → log.exception + sleep — a transient Redis/Postgres outage degrades to retrying ticks, never kills the process.

7. Retry, Backoff & Attempt Accounting

app/retry.py · app/repository.py · app/api/routes.py:retry_job

Attempt semantics (interviewers love this)

  • attempts counts completed attempts and is incremented inside the terminal-transition UPDATE (attempts = Job.attempts + 1 as a SQL expression, not read-modify-write in Python — so it's race-free under the status guard).
  • max_attempts = 4 ⇒ 1 initial attempt + 3 retries (matching the assignment's "3 attempts" retry requirement).
  • In schedule_retry_or_fail, n = job.attempts + 1 is "the attempt that just ended" (the ORM object still holds the pre-increment value). n >= max_attempts → permanent failed.

Backoff schedule: fixed table, not a formula

retry_backoff_schedule = [0, 30, 120]   # seconds
backoff_delay(n) = schedule[min(n-1, len-1)]   # clamps past the end
  • delay 0 → retry_to_pending (status→pending, sync flag→false, started_at→NULL) then immediate re-XADD to the job's priority stream.
  • delay > 0 → retry_to_scheduled with scheduled_at = now + delay, parked in the ZSET; the normal ticker promotion path delivers the retry. Retries reuse the scheduled-jobs machinery instead of a second delay mechanism.
  • Why not exponential+jitter: with only 4 attempts a 3-entry table has the same shape with zero tuning; DECISIONS.md explicitly flags thundering-herd as the accepted trade-off.

One shared path for every failure kind

schedule_retry_or_fail is called from three places with different error payloads: handler exception ({"type": ExcName, "message": ...}), handler timeout ({"type": "HandlerTimeout"}), and reaper-detected worker death ({"type": "WorkerLost", "message": "reclaimed by reaper"}). Crash-retries therefore get identical attempt-counting/terminal semantics as ordinary failures — no drift between two code paths.

Manual retry endpoint

POST /jobs/{id}/retry → reset_failed_to_pending: guarded WHERE status='failed', and it resets attempts=0, clears error/started_at/completed_at, flips the sync flag — a manual retry grants a full fresh budget rather than instantly re-failing at max. Non-failed jobs → 409, unknown → 404.

Broad except Exception in process_job is intentional and annotated (noqa: BLE001): any handler or payload-validation error is treated as retryable. That means a permanently malformed payload burns all 4 attempts before landing in failed — there's no "non-retryable error" classification and no dead-letter queue (acknowledged gap, §19).

8. Crash Recovery — Two Layers for Two Failure Modes

app/ticker/runner.py:reap_stale, _reap_one · app/worker/timeout.py · DECISIONS.md §2

Layer 1 — hung handler, process alive: thread timeout + process recycle

  • run_with_timeout runs each handler in a single-use ThreadPoolExecutor(max_workers=1) and waits future.result(timeout=45). On timeout it raises HandlerTimeout and calls executor.shutdown(wait=False) — the thread is abandoned, because Python cannot kill a thread.
  • The worker routes the job through the normal retry path, then increments a timeout counter; at max_handler_timeouts_before_recycle (default 1) it logs worker.recycling and exits with code 1. Docker's restart: on-failure brings up a clean process — the leaked thread's resources are reclaimed by process death. This is why the worker is the only service with a nonzero exit path.

Layer 2 — dead process: the reaper

  • Runs inside the ticker every reaper_interval_s=30s. For each stream it cursors through XAUTOCLAIM(min_idle_time=visibility_timeout_s*1000, count=100) until the cursor returns "0-0", claiming stale PEL entries as consumer "reaper".
  • Invariant handler_timeout (45s) < visibility_timeout (60s) (boot-time validation) guarantees the in-process layer always resolves a hung-but-alive job first — the reaper only ever sees messages from truly dead processes.

_reap_one — a case analysis of every possible row state

Row state when reclaimedActionWhy
completed / failed / cancelledjust XACK"Ghost" message — e.g. the worker finished but died before acking, or lost the complete-race. Nothing to redo.
processingschedule_retry_or_fail(WorkerLost), then XACKThe classic crash-mid-job: retry with backoff or terminal-fail, same as any failure.
pending/scheduled, synced=falsefinish the Redis handoff inline (XADD or ZADD), mark_synced, XACKWorker won a retry transition then died before re-enqueueing. Recovery is immediate rather than waiting for the 60s reconciler — and attempts are deliberately not touched (the retry decision was already made).

Q: Why timeout-based recovery instead of heartbeats?
DECISIONS.md §5 is candid: heartbeats (Redis TTL key or last_seen_at refreshed well under the timeout) would make detection latency scale with real liveness instead of the slowest job's worst case, and would stop punishing legitimately-slow jobs. Skipped for time; the fixed timeout plus the strict 45<60 invariant is the accepted simpler version.
Q: Reaper on a stream whose group doesn't exist yet?
XAUTOCLAIM raises NOGROUP; reap_stale calls ensure_group per stream first, mirroring every other consumer-group reader. There's a regression-style comment in the code explaining exactly this.

9. Cancellation — Guarded Transitions + Cooperative Polling

app/api/routes.py:cancel_job_route · app/repository.py · app/worker/context.py · app/jobs/handlers.py

The API state machine (bounded retry loop)

for _ in range(3):                       # legal processing<->pending flap
    if cancel_pending_or_scheduled():    # CAS: pending|scheduled → cancelled
        ZREM jobs:delayed                # harmless no-op if not parked
        return 200
    if request_cancel():                 # CAS: processing → set cancel_requested_at
        return 202                       # accepted, cooperative
    job = get_job()
    if None: 404
    if cancelled: return 200             # idempotent repeat-cancel
    if completed|failed: 409
    # else state flapped mid-check → loop
raise 409 "job state is changing; retry"
  • 200 vs 202 distinction: 200 = cancelled outright (was pending/scheduled); 202 = cancellation requested on a processing job — it only takes effect if the handler cooperates.
  • The loop exists because a job can legally move pending→processing (or processing→pending on retry) between the two CAS attempts; three iterations bound the retry instead of spinning.
  • Cancelling a scheduled job also ZREMs it from the delayed set; and even if the ZREM is missed, promote_due drops ids whose row is no longer scheduled (priority lookup misses / DB CAS WHERE status='scheduled' fails) — two independent backstops.

Worker side — PgJobContext (the most nuanced class in the repo)

Handlers receive a ctx implementing set_progress(pct) / cancelled() → bool (a Protocol, so tests can inject fakes). The Postgres-backed implementation:

  • Rate-limited: polls the DB at most once per cancel_poll_interval_s=2s using injected time.monotonic (immune to wall-clock jumps; injectable for tests); between polls returns the cached answer.
  • Coalesced write: when progress changed since last write, one round trip does everything: 
    UPDATE jobs SET progress=:pct
WHERE id=:id AND status='processing'
RETURNING cancel_requested_at
    — writes progress, reads the cancel flag, and confirms the row is still processing in a single statement. When progress hasn't changed it does a cheap flag-only SELECT instead.
  • Lost ownership counts as cancelled: if the row is gone or no longer processing (reaper took it), cancelled() returns True — the handler stops doing work it no longer owns.
  • Own short-lived session per poll: it runs inside the timeout thread, and SQLAlchemy sessions aren't thread-safe — hence the injected session_factory rather than sharing the worker's session.

Handler side

Cancellation is purely cooperative; only handle_batch polls, checking ctx.cancelled() before each item and raising JobCancelled(summary) — an exception that carries the partial summary so the worker can persist partial results: cancel_job(id, summary) does a guarded processing→cancelled with result=summary. Email/webhook/report handlers never observe cancellation mid-run (they're 1–5s sleeps; acceptable).

10. Idempotency — Hash Comparison + Partial Unique Index + Race Path

app/idempotency.py · app/api/routes.py:submit_job · migration 0005

  • Canonical hash: SHA-256 over json.dumps({"type": ..., "payload": ...}, sort_keys=True, separators=(",",":")) — key order and whitespace can't produce false mismatches. Hashing the submitted payload (pre-JSONB) avoids JSONB normalization surprises.
  • Same key + same hash → replay: return the existing job with 200 (vs 202 for a fresh create — the status code itself tells the client it was a replay).
  • Same key + different hash → 409 "idempotency key reused with a different payload" — silently returning a different job would mask a client bug.
  • Uniqueness enforcement: a partial unique index WHERE idempotency_key IS NOT NULL — NULL keys never collide (though Postgres unique treats NULLs as distinct anyway, the partial index also keeps it small).
  • The race path (check-then-insert TOCTOU): two concurrent submissions with the same new key both pass the SELECT check; one INSERT wins, the loser gets IntegrityError → session.rollback() → re-fetch by key → same replay-or-409 logic. The DB constraint is the real arbiter; the pre-check is just the fast path. Tested via monkeypatching (test_race_path_different_payload_conflicts).
  • ≥24h retention requirement: satisfied trivially — keys live on the job row forever (no TTL/cleanup job; acknowledged as unbounded growth in §19).

11. Batch Jobs — Composition over Duplication

app/jobs/handlers.py:handle_batch · app/schemas/payloads.py

  • Batch items are a discriminated union (Field(discriminator="type")) of the three real payload models — an item is validated by exactly the same model as a top-level job, and handle_batch dispatches through the same run_handler registry. Nested batches are structurally impossible (the item union excludes BatchPayload; tested).
  • MAX_BATCH_ITEMS = 500 caps payload size at the schema layer.
  • "Permissive batch" pattern: per-item try/except; failures are collected into {"errors": [{"index", "error"}]}, successes into results, and the parent job completes even if every item failed — item failures are data, not job failure. (Trade-off: batch items get no per-item retry.)
  • Progress: ctx.set_progress(int((i+1)/n*100)) after each item, with an if n else 100 guard for empty batches (div-by-zero). The worker calls init_progress (progress=0) before running and forces progress=100 on completion so tiny batches that finish between polls still end at 100 (tested: test_tiny_batch_reaches_progress_100_without_polling).
  • The deferred import inside handle_batch (from app.jobs.registry import run_handler) breaks the registry↔handlers circular import — a detail worth being able to explain.

12. Keyset (Cursor) Pagination — Not OFFSET

app/cursor.py · app/repository.py:list_jobs

  • Cursor = urlsafe-base64 of {"c": created_at ISO, "i": job_id}. Opaque to clients but transparent/debuggable server-side; not signed (see §16).
  • Query uses a row-value comparison: WHERE (created_at, id) < (:c, :i) ORDER BY created_at DESC, id DESC — SQLAlchemy tuple_() compiles to Postgres's native row comparison, which is index-navigable on ix_jobs_created_at_id.
  • id is the tiebreaker because created_at has microsecond precision but is not unique — without it, rows sharing a timestamp could be skipped or duplicated across pages.
  • limit+1 trick: fetch one extra row; its presence means "there's a next page", and the cursor is built from the last returned row. No expensive COUNT.
  • Why keyset over OFFSET: O(page) instead of O(skip) scan cost, and stable under concurrent inserts (no drifting pages).
  • Garbage cursors: decode_cursor catches ValueError/KeyError/JSONDecodeError and re-raises ValueError, which the route maps to 422 "invalid cursor" — malformed input can't 500.

14. Monitoring — /health, /stats, Structured Logging

app/observability.py · app/core/logging.py · app/api/routes.py

GET /health (readiness)

check_readiness pings Postgres (SELECT 1) and Redis (PING) in independent try/excepts — one backend's failure never masks the other's status. Any failure → 503 with per-check detail ({"postgres": "ok", "redis": "error"}). This endpoint doubles as the Docker healthcheck.

GET /stats — the nuanced one

  • Single pipeline (non-transactional) for all Redis introspection: XINFO GROUPS ×3, XINFO CONSUMERS ×3, ZCARD — one round trip, results sliced positionally.
  • pipe.execute(raise_on_error=False) + _tolerate_nogroup: a stream/group that doesn't exist yet returns NOGROUP as an in-band error object → treated as empty; any other error re-raises and the route converts it to 503. Distinguishing "not created yet" from "broken" is the point.
  • depth vs in_flight: depth = group lag (entries added but never delivered), in_flight = pending (delivered, unacked — the PEL). A comment documents that lag is only nil after XDEL, which this system never issues, so the null fallback is purely defensive.
  • Live worker counting (live_worker_count): a worker busy on the high stream looks idle on normal/low, so per-stream idle is misleading — the function takes each consumer's minimum idle across all streams and counts it live if under visibility_timeout_s. Pure function, unit-tested.
  • Reaper exclusion: XAUTOCLAIM registers "reaper" as a consumer even when it claims nothing, so it would inflate the worker count every 30s; it's filtered by name before counting.
  • DB side: GROUP BY status zero-filled to all six statuses (zero_fill_status_counts — clients always see every key), plus oldest_pending_age_seconds (min created_at of pending → age; the single best queue-latency SLI here).

Logging (structlog)

  • JSON to stdout; shared processor chain (ISO timestamps, level, exc formatting) applied to both structlog-native and foreign stdlib records via ProcessorFormatter + foreign_pre_chain — uvicorn's loggers are hijacked (handlers cleared, propagate=True) so everything, including access logs, comes out as uniform JSON.
  • Job context via contextvars: the worker binds consumer once at startup, then wraps each message in bound_contextvars(job_id=..., message_id=..., stream=...) — every log line inside a job (any module, any depth) automatically carries the job id; the context manager guarantees cleanup between jobs. This satisfies (and is how) the "structured logging with job context" rule from CLAUDE.md — zero print statements in the codebase.
  • Event names are machine-greppable verbs: job.received, job.completed, job.retry_scheduled, job.complete_lost_to_reaper (critical!), worker.recycling, ticker.promoted, ticker.reaped, ticker.reconciled.

15. Performance Details

  • Redis pipelining everywhere multiple commands are issued: promotion XADDs, stats introspection. transaction=False — batching for round-trip elimination, not atomicity (atomicity isn't needed; ordering + claim guard handle safety).
  • Bulk SQL: priorities fetched with one IN query; promotion status flips in one bulk UPDATE; multi-member ZREM.
  • Blocking reads, not polling: idle workers park in XREADGROUP block=5000 — near-zero idle load, ≤5s shutdown-signal latency (the block must expire before the stop flag is checked — that's the real reason block_ms isn't huge).
  • Ticker drain loop (§6) removes the artificial 100 jobs/sec promotion ceiling under burst.
  • Partial indexes keep hot scans (unsynced sweep, idempotency lookup) small and constant-time-ish regardless of table growth.
  • Keyset pagination — no OFFSET scans, no COUNT (§12).
  • Connection hygiene: pool_pre_ping=True (stale pooled connections detected before use — matters with containers restarting around each other); expire_on_commit=False (ORM objects usable after commit without a surprise reload SELECT — necessary because e.g. process_job reads job.attempts after commits); Redis client has explicit socket_connect_timeout=5 / socket_timeout=10 so a dead Redis fails fast instead of hanging a worker.
  • Coalesced progress writes (§9): batch progress costs at most one small UPDATE per 2s, not one per item.
  • Deliberate non-optimizations: count=1 per read (fairness > throughput); one job at a time per worker process (isolation > density); scaling is horizontal via docker compose up --scale worker=N.

16. Security

  • Container hardening: non-root appuser (uid 1000); slim base image; no dev dependencies in the production image (--no-dev); tests/docs excluded via .dockerignore.
  • Input validation as attack-surface control: every payload passes a Pydantic discriminated union with explicit max lengths — to ≤ 320 (RFC email max), subject ≤ 500, body ≤ 20 000, url ≤ 2048, method ≤ 10, report_type ≤ 100, batch items ≤ 500. Unknown job types are rejected by enum coercion (JobType(job_type) raises). This bounds JSONB row size and blocks payload-bloat DoS at the schema layer. (The recent commit dcfb4a4 "fixed security" corresponds to adding these caps.)
  • SQL injection: everything is bound-parameter — ORM expressions or text() with named params (PgJobContext). No string-formatted SQL anywhere.
  • Idempotency conflict = integrity control: the 409-on-different-payload prevents a key-reuse from silently mapping to another payload's job.
  • Error hygiene: malformed cursors → 422 not 500; stats backend failures → clean 503; NOGROUP tolerated. Internal exceptions never leak stack traces through the API surface.
  • Secrets posture (honest answer): jobs/jobs Postgres credentials are hardcoded in docker-compose — acceptable for a local assessment stack, would be env/secret-store injected in production. There is no authentication or authorization on the API (anyone can submit/cancel/retry), no rate limiting, and cursors are unsigned base64 (a client can forge one — worst case they page arbitrarily, no data they couldn't already reach). All acknowledged trade-offs to state proactively.
  • Redis/Postgres are only reachable on the compose network + published localhost ports; no TLS between services (local scope).

17. Testing Strategy — 60 unit + 80+ integration tests

The split

  • Unit tests (fast, no I/O): pure logic — backoff table math (including clamp-past-end), canonical hash stability across key order, cursor round-trip + garbage rejection, payload validation matrix, PgJobContext poll/cache/coalesce behavior via a fake subclass with an injected fake clock, live_worker_count min-idle logic, zero-fill, timeout wrapper (fast/slow/exception propagation), config invariant rejection, logging JSON shape via capsys.
  • Integration tests (real backends): testcontainers spins up real postgres:16 and redis:7 — the exact images from compose. No fakeredis, no SQLite: a previous commit (cebf36e "Removed fakeredis") removed the fake precisely because consumer-group/XAUTOCLAIM semantics differ. Concurrency guards are meaningless against fakes.

Fixture architecture (conftest.py)

  • Containers are session-scoped (started once, ~seconds saved per test), while cleanliness is per-test: TRUNCATE TABLE jobs + FLUSHDB in teardown — fast reset without container churn.
  • Migrations run via Alembic against the test DB (command.upgrade(cfg, "head")) — tests exercise the real migration chain, not create_all, so schema drift between models and migrations is caught (plus explicit test_migration.py assertions on indexes/columns/partial-unique behavior).
  • The client fixture teardown flushes Redis through a fresh client because some tests swap app.state.redis for a broken client to simulate outages — teardown must not depend on what a test left behind.
  • create_app(settings) and run_forever(settings, stop=...) take injected settings/stop-callbacks — the production entrypoints are testable without env vars or signals. The worker/ticker loops are run in-process in tests with a stop closure that flips after N iterations.

19. Known Gaps & Likely "What If" Probes (with prepared answers)

ProbeHonest answer
"Is this exactly-once?"No — at-least-once execution with at-most-once concurrent execution. The only double-run window: a live-but-slow worker exceeding visibility timeout races the reaper; detected and logged critical (job.complete_lost_to_reaper), mitigated by the 45<60 boot-validated invariant.
"Scale the ticker to 2?"Unsafe as-is: concurrent reapers/promoters would double-process (double-XADD is absorbed by the claim guard, but reap-retry accounting could double-increment attempts). Needs leader election or a Redis lock; single instance is fine because ticker workload doesn't scale with job volume.
"Redis loses everything?"AOF (everysec) bounds loss to ~1s. Total loss: manual runbook — recreate groups (automatic on restart), UPDATE ... is_synced_to_redis=FALSE to re-arm the reconciler, manual triage of processing rows. Not automated because auto-detecting "Redis is empty but shouldn't be" risks mass re-enqueue on a false positive.
"Poison message / malformed payload?"Burns all 4 attempts (validation errors are retryable by design) then fails permanently with the error stored. No dead-letter queue — the failed status + error JSONB is the de-facto DLQ, queryable via GET /jobs?status=failed.
"Priority change between fail and retry?"Retry re-reads job.priority from the row at retry time — routed to the current priority stream. But priority isn't mutable via API anyway.
"Handler that ignores cancellation?"Runs to completion; cancel stays a no-op flag (202 was honest about that). Only recourse is the handler timeout ceiling.
"Why does manual retry reset attempts to 0?"Deliberate: a human clicking retry wants a fresh budget; keeping attempts=4 would immediately re-fail. Guarded so only failed jobs qualify (409 otherwise).
"Unbounded growth?"Yes, three places: jobs table (no archival/TTL), idempotency keys (kept forever — exceeds the 24h requirement but never pruned), and stream entries (acked but never XDEL'd/XTRIMed — this is also why lag is reliable). Production fix: partitioning/archival job + XTRIM with the lag caveat.
"Per-job retry policy?"Not supported — max_attempts/schedule are global Settings; max_attempts is per-row in the schema, so the extension point exists (add field to JobSubmission), just not wired.
"Batch item retry?"Items don't retry individually (permissive-batch pattern). A design doc exists (2026-07-01-batch-items-as-real-jobs-design.md) exploring items-as-real-jobs; consciously not implemented.
"What would you change with more time?"DECISIONS.md §5: heartbeat-based liveness instead of fixed visibility timeout — per-job-accurate crash detection, doubles as orchestration liveness signal, at the cost of periodic writes per in-flight job.
"Why sync SQLAlchemy/redis in an async framework?"Endpoints are def (not async def) — FastAPI runs them on its threadpool, so blocking drivers don't stall the event loop. One consistent sync model across API/worker/ticker beat a mixed async/sync codebase for this scope.
